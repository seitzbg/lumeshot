# Porting map — Swift type → ShareX reference

The ShareX repo (`~/git/sharex`, read-only) is the behavioral spec. When
behavior is unclear, read the reference class before inventing semantics.
The repowise MCP index and `sharex-audit-digest.txt` in that repo locate
things fast.

| Swift (this repo) | ShareX reference | Notes |
|---|---|---|
| `SXCore/AppSettings` | `ShareX/ApplicationConfig.cs`, `TaskSettings.cs` | Tiny M1 subset; grows per milestone. Includes `HotkeySettings`/`HotkeyCombo` (Carbon keyCode + modifier mask), defaulted to ⌥⇧3/4/5 (keyCodes 20/21/23, modifiers 2560 = option+shift) to avoid colliding with the system ⌘⇧3/4/5 screenshot shortcuts |
| `SXCore/SettingsStore` | `ShareX.HelpersLib` `SettingsBase.cs` | Corrupt-file backup replaces ShareX's silent error swallowing. `SettingsLoadIssue` has three cases — `.corruptBackedUp`, `.corruptBackupFailed` (backup itself failed, corrupt file left in place), `.readFailed` (file exists but unreadable) — all three are logged and surfaced as a notification by `AppDelegate.handleLoadIssue` |
| `SXCore/NameParser` | `ShareX.HelpersLib` `NameParser.cs` | M1 tokens: %y %mo %d %h %mi %s %ms %rn %ra %width %height %pn %i. `%n` intentionally omitted — verify ShareX semantics before adding in M2 |
| `SXCore/CaptureArtifact` | `ShareX` `ImageInfo`/capture result payload | Plain value type carrying PNG bytes + dimensions + capture time + owning app name between the capture layer and the after-capture pipeline |
| `SXCore/AfterCapturePipeline` | `ShareX` `WorkerTask.cs`, `AfterCaptureTasks` enum | M1 chain: save → clipboard → notify, driven by the `PipelineEffects` protocol (file I/O, clipboard, notifications) so the pipeline is unit-testable without AppKit; upload chain lands in M2 |
| `SXCapture/CapturePermission` | (none — macOS TCC concept) | Wraps `CGPreflightScreenCaptureAccess`/`CGRequestScreenCaptureAccess` and the System Settings deep-link; ShareX/Windows has no screen-recording consent model |
| `SXCapture/ImageEncoder` | `ShareX.HelpersLib` `ImageHelpers.cs` (PNG encode path) | Thin wrapper over `ImageIO`/`CGImageDestination` |
| `SXCapture/DisplayCapture` | `ShareX.ScreenCaptureLib` `Screenshot.cs` | GDI BitBlt → SCScreenshotManager. Captures every display, matching each `SCDisplay` back to its `NSScreen` for backing scale/frame; falls back to scale 2 and a synthesized frame if no match is found |
| `SXCapture/WindowCapture`/`WindowFilter` | `ShareX.ScreenCaptureLib` `WindowsList.cs`, `Screenshot_Window.cs` | EnumWindows → SCShareableContent. `WindowFilter.selectable` drops non-normal-layer, off-screen, sub-50px, and unowned/self windows, sorted largest-first so picker hit-testing (last match wins) resolves to the smallest window under the cursor. Capture resolves the target window's own display scale rather than assuming the primary display's |
| `SXCapture/CaptureGeometry` | `ShareX.ScreenCaptureLib` `CaptureHelpers.cs` | Drag-to-rect normalization and view-point → pixel-crop-rect scaling/clamping shared by the region overlay |
| `SXApp/RegionOverlay` (`RegionOverlaySession`) | `ShareX.ScreenCaptureLib` `RegionCaptureForm.cs` | Freeze-frame model; single-display selection in M1. One borderless `.screenSaver`-level window per display; Esc cancels; a click without a ≥4pt drag is ignored (overlay stays up); drag-release crops and completes |
| `SXApp/WindowPickerSession` | `ShareX.ScreenCaptureLib` `RegionCaptureForm.cs` (window-hover mode) | Transparent full-screen overlays per screen; hover highlights the window under the cursor with an app/title label, click captures, Esc cancels |
| `SXApp/HotkeyManager` | `ShareX.HelpersLib` `HotkeyManager.cs` | RegisterHotKey → Carbon `RegisterEventHotKey`/`InstallEventHandler`. Resolves its Carbon-callback target through a weak `current` static (not an unretained pointer) so a deallocated instance degrades to a no-op instead of a dangling dereference |
| `SXApp/PermissionOnboardingController` | (none — macOS TCC concept) | `isGranted()` for a side-effect-free check, `ensurePermission()` to prompt + show the onboarding window (System Settings deep-link, Relaunch) when not yet granted |
| `SXApp/AppPipelineEffects` | `ShareX` `ClipboardHelpers.cs`, toast notifications | Implements `PipelineEffects` against real `NSPasteboard`/`UNUserNotificationCenter`; notification click reveals the saved file in Finder. Notifications are inert when not running from a bundle (bare `swift run` has no bundle identifier) |
| `SXApp/CaptureCoordinator` | `ShareX` `TaskManager.cs`/capture dispatch | `init(settingsStore:effects:)`. Fullscreen/region/window entry points, each preflighting permission via `PermissionOnboardingController`. Region and window capture guard against double-trigger with an in-flight flag plus a "session already open" check, since a stray second hotkey press or menu click while an overlay is up must not spawn a second overlay. `deliver(image:appName:)` returns `Bool` (false on PNG-encode or pipeline failure) so callers (including the CLI `--capture` debug hook) can react to delivery outcome. Fullscreen captures every display, delivering one artifact each; the last display's capture wins the clipboard |
| `SXApp/StatusItemController` | `ShareX` `TrayIcon.cs` | Menu-bar `NSStatusItem` with the capture menu |
| `SXApp/AppDelegate.registerHotkeys` | `ShareX` `HotkeyManager.cs` callers | Wires the three `AppSettings.hotkeys` combos to the coordinator's capture entry points at launch (⌥⇧3/4/5 → fullscreen/region/window) |
| `SXApp/AppLog` | (macOS-specific, no ShareX equivalent) | Tees diagnostics to `NSLog` and a file at `~/Library/Logs/ShareX-Mac.log`, because a menu-bar app launched from Finder (not a terminal) has no visible stderr; this file is the primary post-hoc diagnostic source (see `docs/smoke-m1.md`) |
| `SXCore/Upload/CustomUploaderConfig` | `ShareX.UploadersLib/CustomUploader/CustomUploaderItem.cs` | PascalCase-keyed `Codable` mirror of the `.sxcu` JSON schema (RequestURL/Headers/Body/Arguments/RegexList/…). `parse(_:)` rejects unsupported `Body` values (e.g. `XML`) before decoding, since the Swift enum can't model them |
| `SXCore/Upload/CustomUploaderEngine` | `ShareX.UploadersLib/CustomUploader/ShareXCustomUploaderSyntaxParser.cs`, `CustomUploaderFunction*.cs` | M2a subset of the token/function language: static request preparation (parameters/headers/arguments) only — no `{input}`/`{prompt}` interactive functions, documented as a known limitation in `docs/smoke-m2a.md` notes |
| `SXCore/Upload/RequestBodyEncoder` | `CustomUploaderItem.cs` body-building (multipart/JSON/form/binary) | Builds the `MultipartFormData`/`FormURLEncoded`/`JSON`/`Binary` request bodies matching `CustomUploaderBody` |
| `SXCore/Upload/ResponseURLParser` | `ShareX.UploadersLib/CustomUploader/Functions/CustomUploaderFunctionResponseURL.cs`, `RegexList` handling | Resolves `$json:path$`/`$regex:n$`/header/literal tokens against the upload response to produce the final URL |
| `SXCore/Upload/UploadDestination`, `UploadSettings` | `ShareX.UploadersLib/UploadersConfig.cs` (`ImageUploaderType`/per-destination settings) | One `UploadDestination` per configured target (custom uploader or Imgur); `activeDestinationID` mirrors ShareX's single "current image uploader" selection. `customUploader.headers`/`.arguments` hold non-secret values only — secret values are the `$keychain$` sentinel (`UploadService.secretSentinel`), resolved at upload time |
| `SXCore/Upload/CredentialStore` (protocol), `SXApp/KeychainCredentialStore` | `ShareX.UploadersLib/UploadersConfig.cs` `[JsonEncrypt]` fields (DPAPI-backed in-place encryption) | Deliberate divergence: instead of encrypting secret fields in place inside `settings.json`, secrets are moved out entirely into the macOS login Keychain at `"<destinationID>/<key>"` (service `org.sharexmac.app`), leaving only the sentinel on disk — closer to "never touches the JSON" than ShareX's DPAPI approach |
| `SXApp/UploadService` | `ShareX/UploadManager.cs`, `TaskManager.cs` upload dispatch | Builds an `Uploader` (Imgur or custom) per destination, rehydrating `$keychain$` sentinels via `CredentialStore` immediately before the request; called as a fire-and-forget task after the local save so upload never gates or blocks it |
| `SXApp/SxcuImporter` | `ShareX/ShareXCLIManager.cs` `.sxcu` argument handling, `CustomUploaderSettingsForm.cs` import | Parses a `.sxcu`, moves header/argument values matching a secret-key heuristic (`Authorization`/`*Token*`/`*Key*`/`*Secret*`, case-insensitive) into the Keychain, and returns an `UploadDestination` with the sentinel in place of each secret |
| `SXCore/History/HistoryStore`, `HistoryEntry` | `ShareX.HistoryLib/HistoryManagerSQLite.cs`, `HistoryItem.cs` | SQLite-backed capture/upload history at `~/Library/Application Support/ShareX-Mac/history.sqlite` — ShareX-parallel storage choice (ShareX also supports a SQLite history manager alongside JSON/XML); one row per capture, updated in place with the resolved URL/failure state once an upload completes or fails |
| `scripts/setup-signing.sh` | (macOS-specific, no ShareX equivalent) | One-time: creates a dedicated `sharex-signing.keychain-db` with a self-signed `sharex-mac-dev` code-signing identity, so the ssh dev loop (`scripts/remote.sh`) can codesign non-interactively (the login keychain is locked to non-interactive ssh sessions) with a *stable* identity — TCC keys the Screen Recording grant off the cert identity, so a stable signature (unlike ad-hoc, which changes every build) means the grant survives rebuilds |
| `scripts/bundle.sh` | (macOS-specific, no ShareX equivalent) | Assembles `dist/ShareX for Mac.app` and signs it: prefers the dedicated dev signing identity from `setup-signing.sh` when its keychain and password file are present, otherwise falls back to ad-hoc signing (`-`) — this is the path CI and any machine without the dev keychain take |
